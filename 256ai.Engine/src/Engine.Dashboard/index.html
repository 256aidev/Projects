Welcome to 256ai Swarm Control

This is the control dashboard for the 256ai Engine — a distributed AI orchestration system that coordinates multiple AI workers (Claude Code instances, Ollama models, and Claude API) across your network. The engine routes tasks to the right worker, tracks their progress, and collects results.

What Can You Do Here?

  • Dashboard tab — See your workers, recent tasks, and activity log at a glance. Send one-off tasks to any worker.
  • Projects tab — Create and manage projects. Projects group related tasks together so you can track progress on larger goals.
  • Task Queue tab — Build a batch of tasks and run them sequentially. Great for multi-step workflows where you want tasks executed one after another.
  • Help tab — You are here. This is the comprehensive reference for everything the system can do.

How It Works (The Short Version)

  1. You type a task objective (like "Write a Python function to parse CSV files").
  2. You pick a domain (general, code, docs, ai-compute, data) that determines which workers can pick it up.
  3. You click Send Task. The engine creates the task with status PENDING.
  4. Workers poll the engine every 5 seconds looking for tasks in their domain.
  5. A matching worker claims the task, executes it, and sends back the result.
  6. You see the result in the Task Viewer panel.
Tip: Click on any task in the Recent Tasks list to view its output, details, and timeline in the Task Viewer panel below.

System Architecture

The 256ai Engine is organized into 5 layers. Understanding these layers helps you understand who does what and where your tasks actually get executed.

The 5 Layers

LayerWho / WhatResponsibility
1. Strategy Human (Mark) Final authority. Approves plans, makes architectural decisions, sets direction. The engine never overrides the Strategy Layer.
2. Control Plane This Engine (port 5100) Routes tasks to workers, validates messages, manages state machines, processes heartbeats, stores results. This dashboard talks to the Control Plane.
3. Execution Nodes Workers (Claude Code, Ollama, Claude API) Execute tasks, send heartbeats, report progress, escalate anomalies. These are the "hands" that do the actual work.
4. Model & Compute Ollama, Qwen, Llama AI inference for applications (not the engine itself). When an app needs AI, it calls Ollama on Dragon or AI02. The engine doesn't use Ollama directly — it orchestrates Claude Code instances.
5. Monitoring This Dashboard, Logs, Metrics Observability layer. You're using it right now. Monitors health, task flow, worker status, and escalations.
Key distinction: The engine orchestrates Claude Code instances as workers, NOT Ollama. Ollama is Layer 4 compute that applications use for inference. Some workers happen to use Ollama as their provider for executing tasks, but the engine treats all workers the same regardless of their backend.

Message Types (HL7-Style)

The engine uses a messaging protocol inspired by HL7 (healthcare messaging standard). Each message type has a 3-letter code:

CodeNamePurpose
AGTAgent TaskTask dispatch — "here's a job for you"
AACAgent AckWorker acknowledges receipt of a task
ARGAgent ResultWorker sends back the completed result
AEXAgent ExceptionWorker raises an escalation (something went wrong)
AHEAgent HeartbeatWorker sends a "still alive" pulse every 15–30 seconds
ADIAgent DispositionHuman resolves an escalation (accept/reject/defer)
AHSApp Health StatusApplication reports its health
ASCSynthetic CheckEnd-to-end test to verify the system works

Task Domains

Every task has a domain that controls which workers can pick it up. When you submit a task, you choose a domain. Workers declare which domains they handle in their configuration. Only workers whose domain list includes your task's domain will claim and execute it.

Available Domains

DomainUse ForWhich Workers Handle It
general Anything that doesn't fit a specific category. This is the default and is the safest choice if you're not sure. All workers (every worker declares general)
code Code generation, refactoring, bug fixes, code review, writing scripts, debugging. Use this when you want code written or analyzed. Workers running Claude Code or other coding-capable models
docs Documentation writing, markdown generation, README creation, text summarization, content writing. Any text-heavy task. Any text-capable worker
ai-compute Heavy AI inference tasks — model inference, embeddings, data processing, anything that benefits from GPU compute and local models. Workers with Ollama/Qwen backends (Dragon, AI02). These machines have GPUs.
data Data analysis, transformation, database queries, CSV processing, ETL tasks, statistics. Workers with data processing capability

How Routing Works

  1. You submit a task with a domain (e.g., code).
  2. The task enters the queue with status PENDING.
  3. Every 5 seconds, each worker polls the Control Plane: "Do you have any tasks in my domains?"
  4. The first worker whose domain list includes code claims the task.
  5. If no workers handle that domain, the task stays PENDING until one comes online.
Important: If you submit a task to ai-compute but no Ollama workers are online, the task will sit in PENDING forever. Check the Workers panel to make sure at least one worker handles your chosen domain.

When to Use What

  • Not sure? Use general. Every worker can handle it, so it will always get picked up.
  • Want code written, reviewed, or debugged? Use code.
  • Want local AI (Ollama/Qwen) to do the work? Use ai-compute.
  • Writing docs or text content? Use docs.
  • Data crunching? Use data.

Sending Tasks

The Send Command section on the Dashboard tab is the quickest way to submit a single task to the swarm. Here's exactly how to use it, step by step.

Step-by-Step

  1. Go to the Dashboard tab.
  2. In the Send Command panel, type your task objective in the text box. Be specific and clear about what you want. For example:
    • "Write a Python function that validates email addresses using regex"
    • "Analyze the login flow in the BaZi app and suggest improvements"
    • "Summarize the key points of this document: [paste content]"
  3. Select a domain using the radio buttons (General, Code, Docs, etc.). See the Task Domains section for guidance.
  4. Click Send Task.
  5. The task will appear in the Recent Tasks list with status PENDING.
  6. A worker will claim it (usually within 5–10 seconds).
  7. Click on the task in the list to watch its progress in the Task Viewer.
  8. When the worker finishes, the result appears in the Output tab of the Task Viewer.

Task Viewer Tabs

When you click a task, the Task Viewer shows three tabs:

TabWhat It Shows
OutputThe actual result/response from the worker. This is what the AI produced.
DetailsMetadata: task ID, domain, status, assigned worker, timestamps, execution mode.
TimelineChronological view: when the task was created, claimed, started, and completed. Useful for debugging slow tasks.
Auto-refresh: When viewing an in-progress task, the Task Viewer automatically refreshes every 2 seconds so you can watch the progress live.

Writing Good Objectives

The quality of your results depends heavily on how you write the objective. Here are some tips:

  • Be specific. "Write code" is vague. "Write a TypeScript function that converts Fahrenheit to Celsius" is actionable.
  • Include context. If the task relates to a specific project or file, mention it.
  • State the expected output. "Return a JSON object with..." or "Write the output as a markdown table" helps the worker format correctly.
  • One task, one objective. Don't combine "write the function AND update the tests AND deploy." Split these into separate tasks (or use the Task Queue for sequencing).

Task Queue

The Task Queue tab lets you build a batch of tasks and execute them one after another, sequentially. This is ideal for multi-step workflows where you need tasks to run in order, or when you want to submit a large batch and walk away while they execute.

How to Use the Task Queue

  1. Go to the Task Queue tab.
  2. Select a domain from the dropdown (general, code, docs, ai-compute, data).
  3. Type a task objective in the text box and press Enter or click Add.
  4. Repeat to add more tasks. They appear in the queue list below.
  5. When your queue is ready, click Start Queue.
  6. The engine submits the first task and waits for it to complete before submitting the next one.
  7. A progress bar shows how many tasks have completed.

Queue Settings

SettingWhat It Does
Auto-resume on failure If a task fails, the queue automatically moves on to the next task instead of stopping. Enabled by default.
Max retries How many times to retry a failed task before moving on. Default: 3.
Retry delay (sec) How many seconds to wait before retrying a failed task. Default: 30 seconds.

Queue Controls

  • Start Queue — Begins processing tasks from the top of the queue.
  • Pause — Pauses after the current task finishes. Does not cancel the running task.
  • Clear — Removes all tasks from the queue. If a task is currently running, it continues to completion but no more tasks are submitted.

Queue Status Badges

BadgeMeaning
IdleQueue is not running. Add tasks and hit Start Queue.
RunningActively processing a task. The current task objective is shown.
PausedQueue was paused. Click Start Queue to resume.
WaitingWaiting for a task to complete before moving to the next one.
Tip: You can add tasks to the queue while it's running. New tasks are added to the end and will be processed when the queue reaches them.

Task Lifecycle

Every task moves through a series of statuses from creation to completion. Understanding this lifecycle helps you diagnose issues and know what's happening with your tasks at any given moment.

Status Flow

PENDING → LEASED → ACKED → RUNNING → COMPLETED
   |          |         |         |
   |          |         |         +--→ FAIL
   +----------+---------+---------+--→ CANCELLED

Status Definitions

StatusWhat It MeansHow Long It Typically Lasts
PENDING Task has been created and is waiting in the queue for a worker to claim it. No worker has touched it yet. 0–10 seconds (depends on worker poll timing)
LEASED A worker has polled and claimed the task, but hasn't acknowledged it yet. Think of it as "reserved." Milliseconds to a few seconds
ACKED The worker has acknowledged receipt (sent an ACK message). The worker is preparing to execute. Milliseconds to a few seconds
RUNNING The worker is actively executing the task. It may report progress updates during this phase. Seconds to minutes (depends on task complexity and AI provider)
COMPLETED The worker finished successfully and submitted a result. The output is available in the Task Viewer. Final state
FAIL The worker tried but the task failed. Check the task details for error information. Final state
CANCELLED The task was cancelled before completion, either by you or by the system. Final state

Cancelling Tasks

You can cancel any task that hasn't reached a final state (COMPLETED, FAIL, or CANCELLED). In the Recent Tasks list, click the orange Cancel button next to the task. The task status changes to CANCELLED immediately.

Note: Cancelling a task in the Control Plane does not immediately stop the worker. The worker may continue executing until it next checks in. The result will be discarded if it arrives after cancellation.

Workers

Workers are the AI agents that actually execute your tasks. Each worker runs on a specific machine, uses a specific AI provider, and handles specific domains. The Workers panel on the Dashboard tab shows all registered workers.

Worker Information

Each worker in the list shows:

FieldWhat It Shows
NameThe worker's unique identifier (e.g., worker-256ai-001). Set in the worker's configuration.
RoleWhat kind of worker it is: claude-code, ollama, or claude-api.
ProviderThe AI backend the worker uses to execute tasks.
IP AddressThe network address of the machine the worker runs on.
StatusOnline (green) if heartbeat received in the last 60 seconds, Offline (red) otherwise.

Worker Providers

ProviderDescriptionBest For
claude-code Runs tasks via Claude Code CLI as a subprocess. Full access to the local filesystem, tools, and development environment. Code generation, file editing, complex multi-step tasks, anything that needs filesystem access
ollama Sends the task objective to a local Ollama model (Qwen, Llama, etc.) via HTTP API. Fast, free, runs on GPU. Quick AI inference, embeddings, data processing, local-only tasks
claude-api Calls the Anthropic Claude API directly. Requires an API key configured on the worker. Cloud-scale tasks, when local compute is saturated, general-purpose AI

Heartbeats and Online Status

Every worker sends a heartbeat to the Control Plane every 15–30 seconds. The heartbeat includes the worker's ID, role, IP address, status, and capacity information. If the Control Plane hasn't received a heartbeat from a worker in the last 60 seconds, that worker is marked Offline.

An offline worker will not receive new tasks. Tasks already claimed by a worker that goes offline may get stuck in LEASED or RUNNING status until the lease timeout returns them to PENDING.

Removing Stale Workers

If a worker is offline and you know it's not coming back (machine shut down, worker decommissioned), you can remove it from the registry. In the Workers list, click the delete button next to the offline worker. This cleans up the dashboard and prevents stale entries from cluttering the view.

Caution: Only remove workers that you're sure are permanently offline. If the worker comes back online, it will re-register itself automatically via its next heartbeat.

Projects

Projects let you group related tasks together under a single name. This is useful when you're working on a multi-step goal and want to track all the tasks associated with it in one place.

Creating a Project

  1. Go to the Projects tab.
  2. Click + New Project.
  3. Fill in the details:
    • Name — A human-readable name (e.g., "BaZi Astrology App").
    • Domain — The project's primary domain category.
    • Description — Brief description of what the project is.
    • Working Dir — The filesystem path where the project's code lives.
    • Spec / MD — Optional. Upload or paste a markdown spec file (from ChatGPT, design docs, etc.).
  4. Click Create Project.

Filtering by Project

Click on a project chip to filter the Dashboard's task list to only show tasks belonging to that project. A filter badge appears showing the active filter. Click the X on the badge to clear the filter and see all tasks again.

Project Task Counts

Each project chip shows a task count in the format completed/total (e.g., 5/8 means 5 of 8 tasks are complete). This updates automatically as tasks are submitted and completed.

API Reference

The Control Plane exposes a REST API on http://10.0.1.147:5100. You can call these endpoints from any tool that speaks HTTP — curl, Postman, PowerShell, scripts, other applications, or Claude Code itself.

Health Endpoints

GET /health/summary
Get overall system health status: OK, DEGRADED, or DOWN. Shows worker counts, app health, synthetic check results, and pending escalations.
GET /health/workers
List all registered workers with their ID, role, provider, IP, status, and last-seen timestamp. Workers are "online" if heartbeat was received within 60 seconds.
POST /health/heartbeat
Worker sends a heartbeat to prove it's alive. Body: { workerId, status, role, ipAddress, version, capacity }. Workers should call this every 15–30 seconds.
DELETE /health/workers/{workerId}
Remove a dead or stale worker from the registry. Use this to clean up workers that are permanently offline.
GET /health/apps
List health status of all managed applications. Shows app ID, environment, status, latency, error rate.
GET /health/synthetic
List synthetic check results. Optional parameter: ?limit=50.

Task Endpoints

POST /tasks
Submit a new task. Required body fields: objective (string), domain (string), expectedOutputs (string). Optional: constraints, inputs, validationCriteria, timeLimitSeconds, projectId, parentTaskId, executionMode, dependsOn.
GET /tasks
List tasks with optional filters. Parameters: ?status=PENDING, ?projectId=xxx, ?parentTaskId=xxx, ?limit=50.
GET /tasks/{id}
Get a specific task by ID. Returns full details including status, assigned worker, timestamps, progress, and result.
GET /tasks/poll
Worker polls for available tasks. Claims the first matching PENDING task. Required: ?workerId=xxx. Optional: &domains=general,code.
POST /tasks/{id}/ack
Worker acknowledges task receipt after polling. Body: { workerId }. Transitions task from LEASED to ACKED.
POST /tasks/{id}/progress
Worker reports progress on a long-running task. Body: { workerId, message, percentComplete, currentStep }.
POST /tasks/{id}/result
Worker submits the task result. Body: { workerId, success, outputs, executionTimeMs }. Transitions task to COMPLETED or FAIL.
POST /tasks/{id}/cancel
Cancel a pending or in-progress task. Sets status to CANCELLED.

Escalation Endpoints

GET /escalations
List escalations (issues raised by workers). Filter: ?disposition=Pending, ?limit=50. Levels: CONCERN, RISK, BLOCKER.
GET /escalations/{id}
Get full details of an escalation: level, evidence, impact, recommendation, disposition status.
PUT /escalations/{id}
Disposition an escalation. Body: { disposition, reason, reviewDate }. Values: Accepted, Rejected (requires reason), Deferred (requires reviewDate).

Event & Benchmark Endpoints

GET /events
Get recent activity events (tasks, workers, escalations). Parameter: ?limit=50.
POST /benchmark/run
Submit a batch of synthetic benchmark tasks. Body: { taskCount, domain, projectId }. Returns a projectId to track results.
GET /benchmark/{projectId}
Get live benchmark results: throughput (tasks/sec, tasks/min), latency percentiles (p50, p95, max), per-worker distribution.

Project Endpoints

GET /projects
List all projects with their name, domain, description, and working directory.
POST /projects
Create a new project. Body: { name, domain, description, workingDir, specMarkdown }.
GET /projects/{id}
Get project details including task counts by status.

MCP Server (Model Context Protocol)

The Control Plane exposes an MCP SSE endpoint that allows Claude Code instances (and any other MCP-compatible client) to interact with the engine natively using tool calls. This means Claude Code can submit tasks, check health, manage escalations, and run benchmarks — all through its native tool-calling interface instead of raw HTTP requests.

Connecting from Claude Code

Add this to your Claude Code MCP configuration (typically in .claude/settings.json or project settings):

{
  "mcpServers": {
    "256ai-engine": {
      "url": "http://10.0.1.147:5100/sse"
    }
  }
}

Once connected, Claude Code will have access to 11 tools it can call directly.

Available MCP Tools

ToolDescriptionExample Usage
submit_task Submit a new task to the engine "Submit a task to write a Python CSV parser in the code domain"
get_task Get status and result of a task by ID "What's the status of task abc123?"
list_tasks List tasks, filter by status or project "Show me all pending tasks" or "List completed tasks for project X"
cancel_task Cancel a pending or in-progress task "Cancel task abc123"
system_health Get overall system health "How's the engine doing?" or "System health"
list_workers List all workers and their status "List workers" or "Who's online?"
remove_worker Remove a stale/dead worker "Remove worker worker-old-001"
list_escalations List escalations from workers "Any pending escalations?"
dispose_escalation Accept, reject, or defer an escalation "Accept escalation abc123" or "Reject escalation abc123 — false positive"
run_benchmark Submit N benchmark tasks "Run a benchmark with 20 tasks"
benchmark_results Get throughput results for a benchmark "Get benchmark results for benchmark-20260216-070000"
How to use: Once the MCP server is connected, just talk naturally to Claude Code. Say things like "submit a task to analyze the login flow" or "list workers" and Claude Code will call the appropriate MCP tool automatically.

Benchmarking

The benchmark tools let you measure the throughput of your swarm — how fast the system can route and complete tasks. This is useful for understanding your system's capacity, identifying bottlenecks, and testing after configuration changes.

How Benchmarks Work

A benchmark submits N synthetic tasks with a simple objective: "Benchmark task X/N — respond with OK." These tasks are intentionally trivial so that execution time is minimal. What you're measuring is the engine's routing throughput: how fast can the Control Plane route tasks to workers and collect results?

Running a Benchmark (PowerShell Script)

The easiest way to benchmark is the PowerShell script:

# Quick 5-task test
.\scripts\benchmark.ps1 -TaskCount 5

# 20 tasks in burst mode (all submitted at once)
.\scripts\benchmark.ps1 -TaskCount 20 -Burst

# Target only code-domain workers
.\scripts\benchmark.ps1 -TaskCount 10 -Domain code

# Dry run: submit tasks but don't wait for completion
.\scripts\benchmark.ps1 -TaskCount 50 -DryRun

Script Parameters

FlagDefaultDescription
-TaskCount10Number of benchmark tasks to submit
-Domain"general"Target domain (controls which workers participate)
-BurstfalseSubmit all tasks at once (vs. one per second)
-TimeoutSec300Max wait time for all tasks to complete
-ControlPlane"http://10.0.1.147:5100"Target Control Plane URL
-DryRunfalseSubmit tasks but don't wait for completion
-ProjectIdauto-generatedCustom project ID for grouping

Running a Benchmark (REST API)

# Submit 20 benchmark tasks
curl -X POST http://10.0.1.147:5100/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"taskCount": 20, "domain": "general"}'

# Check results (use the projectId from the response above)
curl http://10.0.1.147:5100/benchmark/benchmark-20260216-070000

Running a Benchmark (MCP)

From Claude Code with MCP connected, just say:

  • "Run a benchmark with 10 tasks"
  • "Get benchmark results for benchmark-20260216-070000"

Understanding Results

MetricWhat It Tells You
Wall timeTotal elapsed time from first task submitted to last task completed.
Throughput (tasks/sec)How many tasks the system completes per second. Higher is better.
Throughput (tasks/min)Same metric, scaled to per-minute for easier human reasoning.
Latency avg/p50/p95/maxHow long individual tasks take from creation to completion. P50 = median, P95 = worst 5% of tasks.
Per-worker distributionHow many tasks each worker handled. Ideally, tasks are spread evenly across workers.
Bottleneck: Workers poll every 5 seconds. With 6 workers, the theoretical maximum routing throughput is about 1.2 tasks/second (6 workers ÷ 5 second poll interval). Actual throughput will be lower due to execution time.

Escalations

When a worker encounters something it can't handle — ambiguous instructions, an error it doesn't know how to fix, a security concern, or any situation where it needs human judgment — it raises an escalation. Escalations surface issues for human review and are a core part of the engine's safety model.

Escalation Levels

LevelSeverityWhen It's Used
CONCERN Low — informational Minor issue that might be worth noting. The worker continued working but flagged something unusual.
RISK Medium — needs attention Potential problem. The task result may be inaccurate or the worker isn't confident in its output.
BLOCKER High — cannot proceed The worker is stuck and cannot continue without human intervention. Requires immediate action.

Dispositions

Every escalation must receive a disposition. This is a golden rule of the engine. You resolve escalations by choosing one of three dispositions:

DispositionMeaningWhen to Use
Accepted Issue acknowledged and resolved You've reviewed it and it's valid. Take whatever action is needed.
Rejected Issue dismissed (requires a reason) False positive, not an issue, or the worker was wrong to escalate. You must provide a reason.
Deferred Will review later (requires a review date) Valid issue but not urgent. Set a date to revisit it.

Managing Escalations via API

# List pending escalations
curl "http://10.0.1.147:5100/escalations?disposition=Pending"

# Accept an escalation
curl -X PUT http://10.0.1.147:5100/escalations/{id} \
  -H "Content-Type: application/json" \
  -d '{"disposition": "Accepted"}'

# Reject with reason
curl -X PUT http://10.0.1.147:5100/escalations/{id} \
  -H "Content-Type: application/json" \
  -d '{"disposition": "Rejected", "reason": "False positive"}'

# Defer for later
curl -X PUT http://10.0.1.147:5100/escalations/{id} \
  -H "Content-Type: application/json" \
  -d '{"disposition": "Deferred", "reviewDate": "2026-03-01T00:00:00Z"}'

Troubleshooting

Common issues and how to resolve them.

Dashboard Shows "DOWN" Status

This means the dashboard can't reach the Control Plane at http://10.0.1.147:5100.

  • Check that the Control Plane is running: curl http://10.0.1.147:5100/health/summary
  • If it returns "connection refused," the Control Plane process needs to be started: dotnet run --project src/Engine.ControlPlane --urls "http://localhost:5100"
  • Check Windows Firewall — port 5100 must be accessible from the machine running the dashboard.
  • Verify the IP address is correct. The Control Plane runs on Dragon (10.0.1.147).

Task Stuck in PENDING

The task was submitted but no worker is picking it up.

  • Check the domain. Are any online workers handling that domain? If you submitted to ai-compute but only general workers are online, it won't get picked up.
  • Check worker status. Are workers actually online? Look at the Workers panel. If all workers are offline, no one is polling.
  • Wait a moment. Workers poll every 5 seconds. Give it 10–15 seconds before worrying.
  • Check worker logs. The worker process may be running but erroring on the poll. Check the worker's terminal output.

Task Stuck in LEASED or ACKED

A worker claimed the task but hasn't completed it.

  • The worker may be executing a long-running task. Check if the worker is still online.
  • If the worker went offline mid-task, the task is stuck. Cancel it and resubmit.
  • Future: Lease timeout will automatically return these to PENDING after 60 seconds (not yet implemented).

Worker Shows "Offline"

  • The worker process may have crashed or been stopped. Check the worker machine.
  • Network connectivity issue — the worker can't reach the Control Plane. Ping 10.0.1.147 from the worker machine.
  • The worker is running but its heartbeat is failing. Check the worker's terminal for HTTP errors.

Task Completed But No Output

  • Click the task in Recent Tasks, then check the Details tab for the raw result JSON.
  • The worker may have returned an empty or malformed result. Check the success field in details.
  • If success: false, check the worker logs for the error that caused the failure.

MCP Not Connecting

  • Verify the SSE endpoint is reachable: curl http://10.0.1.147:5100/sse (should start streaming).
  • Check your Claude Code MCP configuration. The URL must be exactly http://10.0.1.147:5100/sse.
  • Make sure the Control Plane was restarted after the MCP code was added. MCP won't work on an old build.

Slow Throughput

  • Expected bottleneck: Workers poll every 5 seconds. With 6 workers, max theoretical throughput is ~1.2 tasks/second.
  • If throughput is much lower than expected, check how many workers are actually online and in the right domain.
  • Ollama workers are slower than Claude API workers for simple tasks. Use claude-api workers for fast echo-type tasks.
  • Run a benchmark to get exact numbers: .\scripts\benchmark.ps1 -TaskCount 20 -Burst

Build Errors After Code Changes

  • If dotnet build fails with file lock errors (MSB3027), the Control Plane is still running. Stop it first, then build, then restart.
  • The compilation may succeed (0 errors) but the copy step fails because the .exe is locked. This is a file-lock issue, not a code issue.
